# open-r1 — LUNAR Training

This directory contains the RLVR training code for LUNAR. It is adapted from the [Hugging Face open-r1](https://github.com/huggingface/open-r1) framework, an open reproduction of DeepSeek-R1's reinforcement learning pipeline. The upstream codebase has been modified to support domain-specific ADRD training with a custom reward function, self-certainty-aware advantage ([Intuitor](https://github.com/sunblaze-ucb/Intuitor/blob/main/open-r1-intuitor/src/open_r1/intuitor_trainer.py)), and task-specific oversampling.

---

## What's Modified from Upstream

The following changes were made relative to the original open-r1 codebase:

- **Custom reward function**: replaced math/code verifiers with a binary verifier that checks whether the model's boxed answer matches the ground-truth ADRD diagnostic label.
- **Self-certainty-aware advantage (SCe)**: augments the standard Dr.GRPO reward advantage with a KL-divergence-based confidence signal. This penalizes overconfident wrong answers and rewards confident correct ones, improving calibration and output conciseness.
- **Task-specific oversampling**: minority diagnostic classes (LBD, VD, PSY, SEF, OTH) are oversampled with replacement to match the majority class size, applied independently per question type before each training epoch.
- **ADRD question types**: training data is formatted as closed-ended multiple-choice questions across six clinical task types — cognitive status (COG), primary etiology (ETPR), amyloid PET (PET), amyloid CSF (CSF), DaT imaging (DAT), and neuropathology (NP-ONE).
- **Evaluation code is not included here.** All evaluation is handled in `adrd_simplified_evaluation/`.

## Data Format

Each training instance is a JSON object with the following fields:

```json
{
  "prompt": "The patient is a ... [clinical narrative]\n\nQuestion: Identify the cognitive status of this patient using the provided information.\nAnswer Choices:\nA. Normal cognition\nB. Mild cognitive impairment\nC. Dementia",
  "answer": "B"
}
```

The prompt is constructed automatically from:
1. A clinical prose narrative generated by Qwen3-32B from harmonized JSON data (see `data_preparation/`)
2. One of six diagnostic question types, randomly assigned per participant
3. Randomly shuffled answer choices

It expects the following fields in the dataframe: `visit_summary`, `question`, `options`.

See `data_preparation/` for how to produce this dataset from raw cohort files.

## Training

LUNAR was trained using the Dr.GRPO algorithm on top of **Qwen2.5-3B-Instruct** with DeepSpeed ZeRO-2, running for 1,491 steps (1 epoch) on 4× NVIDIA H200 GPUs (~9 hours).

All training hyperparameters — including learning rate, KL penalty, clip epsilon, SCe weight (λ), batch size, generation temperature, max token lengths, and optimizer settings — are defined in the config file:

```
recipes/Qwen2.5-3B-Instruct/grpo/config_name.yaml
```

Update that file before launching training.

### Launch

```bash
ACCELERATE_LOG_LEVEL=info accelerate launch \
  --config_file recipes/accelerate_configs/zero2.yaml \
  --num_processes=4 \
  src/open_r1/grpo.py \
  --config recipes/Qwen2.5-3B-Instruct/grpo/config_name.yaml
```

For Slurm clusters:

```bash
cd src
sbatch --job-name=job_name \
  --nodes=1 \
  --time=24:00:00 \
  ../slurm/train.slurm \
  --model Qwen2.5-3B-Instruct \
  --task grpo \
  --config name \
  --accelerator zero2 
```

## Ablations

The paper includes four training variants. Select the appropriate config to reproduce each:

| Variant | Config | Description |
|---------|--------|-------------|
| `LUNAR` | `config_lunar.yaml` | Full model — oversampling + SCe advantage |
| `LUNAR-OS` | `config_lunar_os.yaml` | SCe advantage only, no oversampling |
| `LUNAR-SCe` | `config_lunar_sce.yaml` | Oversampling only, standard reward advantage |
| `LUNAR-OS-SCe` | `config_lunar_os_sce.yaml` | Vanilla Dr.GRPO — no oversampling, no SCe |

Each config follows the same structure as `config_lunar.yaml`. Toggle oversampling and the SCe advantage using the relevant flags documented inside the config files.

## SFT Baseline

A supervised fine-tuning (SFT) baseline is provided for comparison. It uses the same backbone and training partition with cross-entropy loss, prompting the model to output only the final boxed answer without chain-of-thought reasoning.

All SFT hyperparameters are defined in:

```
recipes/Qwen2.5-3B-Instruct/sft/config_lunar_sft.yaml
```

Launch with:

```bash
ACCELERATE_LOG_LEVEL=info accelerate launch \
  --config_file recipes/accelerate_configs/zero2.yaml \
  --num_processes=4 \
  src/open_r1/sft.py \
  --config recipes/Qwen2.5-3B-Instruct/sft/config_lunar_sft.yaml
```

## Troubleshooting

- **vLLM version mismatch**: PyTorch 2.6.0 and vLLM 0.8.5.post1 are compiled together — mismatched versions will cause runtime errors. Install them as specified above.
- **Flash-attention build failures**: ensure `setuptools` is installed before running `flash-attn --no-build-isolation`.
- **OOM errors**: reduce `per_device_train_batch_size` or the number of completions per case in the config. The paper used 16 completions per case across 4× H200 GPUs.
- **HuggingFace cache on shared clusters**: set cache paths to avoid filling home directories:
  ```bash
  export HF_HOME=/path/to/shared/cache
  export TRANSFORMERS_CACHE=/path/to/shared/cache
  ```
